= HTTP
:toc: right
:imagesdir: http/images

Every client interaction with Enonic XP — browser request, API call, admin tool, websocket upgrade — flows through the same HTTP gateway. This page covers the common request pipeline that every request passes through, the four services XP dispatches to, and the primitives (request/response shapes, filters, error handlers, websockets, SSE, ID providers) that make up the HTTP layer.

== Services

XP listens on `port 8080` (configurable) and dispatches requests to one of four *services*, each with a fixed URL prefix and a distinct role:

[cols="1,2,3", options="header"]
|===
| Service | Prefix | Use it for
| <<apis#, API>> | `/api` | Custom HTTP APIs, microservices, headless integrations.
| <<admin#, Admin>> | `/admin` | XP administration UIs (Content Studio is itself an admin tool) and extensions that mount inside them.
| <<sites#, Site>> | `/site` | CMS-driven websites: pages, parts, layouts, content rendering.
| <<webapps#, Webapp>> | `/webapp` | Standalone web applications — served by your app, no CMS context.
|===

A single app can contribute to any number of these services simultaneously — a CMS site, a webapp dashboard, a custom API, and an admin tool — all from the same bundle.

== Request flow

Every request, regardless of which service ultimately handles it, passes through a common pre-processing pipeline before reaching your code:

image::common-pipeline.svg[Common HTTP request pipeline, 768px]

Vhost filters:: Request routing based on <<deployment/vhosts#, virtual hosts>>.

Platform filters:: Optional system level request filters.

Autologin filter:: The <<deployment/vhosts#, virtual host contextual ID provider>> is invoked, unless a user already exists in the context.

Service pipeline:: The pipeline of the matching service is executed (API invocation, admin tool routing, site rendering, or webapp dispatch).

Auth handler:: Triggered if the pipeline returns a `401 Unauthorized`. The contextual <<http/id-providers#, ID provider>> is invoked and may handle the request.

Error handler:: If the pipeline returns an unhandled error (everything but `200 OK`) that was not handled by an ID provider, the contextual <<http/error#, error handler>> kicks in.

Platform response filters:: Optional system level response filters.

Standard error handler:: Finally, if the request fails with an error that is still unhandled, XP's standard error handler kicks in.

== Pages in this section

<<http/request-response#, Request & Response>>:: The Request and Response object shapes — headers, body, cookies, status codes — and the way XP populates and reads them.

<<http/filters#, Filters>>:: Reusable wrappers that run before and after HTTP functions in the pipeline. Used for cross-cutting concerns: auth gates, logging, caching, response rewriting.

<<http/error#, Error handler>>:: Custom error pages and exception-to-response mapping. Replaces the default stack-trace response.

<<http/websocket#, WebSocket>>:: Two-way communication channels. Any HTTP function can upgrade an incoming request into a websocket session by returning a `webSocket` object.

<<http/sse#, SSE>>:: Server-Sent Events streams. One-way streaming from server to client over a long-lived HTTP response.

<<http/id-providers#, ID providers>>:: Pluggable authentication that hooks the request pipeline — `handle401`, `autoLogin`, `login`/`logout` endpoints, and ID-provider-as-HTTP-function.
