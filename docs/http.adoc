== Request flow

The XP runtime serves HTTP requests on `port 8080` (customizable). Every request — regardless of which service ultimately handles it — passes through a common pre-processing pipeline before reaching your code:

image::common-pipeline.svg[Common HTTP request pipeline, 768px]

Vhost filters:: Request routing based on <<deployment/vhosts#, virtual hosts>>.

Platform filters:: Optional system level request filters.

Autologin filter:: The <<deployment/vhosts#, virtual host contextual ID provider>> is invoked, unless a user already exists in the context.

Portal service:: The pipeline of the matching <<portal#, Portal service>> is executed (api invocation, webapp dispatch, site rendering, or admin tooling).

Auth handler:: Triggered if the pipeline returns a `401 Unauthorized`. The contextual ID Provider is invoked and may handle the request.

Error handler:: If the pipeline returns an unhandled error (everything but `200 OK`) that was not handled by an ID Provider, the contextual <<http/error#, error handler>> kicks in.

Platform response filters:: Optional system level response filters.

Standard error handler:: Finally, if the request fails with an error that is still unhandled, XP's standard error handler kicks in.


== Pages in this section

<<http/request-response#, Request & Response>>:: The Request and Response object shapes — headers, body, cookies, status codes — and the way XP populates and reads them.

<<http/filters#, Filters>>:: Reusable wrappers that run before and after HTTP functions in the pipeline. Used for cross-cutting concerns: auth gates, logging, caching, response rewriting.

<<http/error#, Error handler>>:: Custom error pages and exception-to-response mapping. Replaces the default stack-trace response.

<<http/websocket#, WebSocket>>:: Two-way communication channels. Any HTTP function can upgrade an incoming request into a websocket session by returning a `webSocket` object.

<<http/sse#, SSE>>:: Server-Sent Events streams. One-way streaming from server to client over a long-lived HTTP response.

For the gateway routing model and per-service pipelines, see <<portal#, Portal>>.
