= Building Admin Extensions
:toc: right
:imagesdir: images

Admin extensions are app-provided UI components that mount into existing <<admin-tools#,admin tools>> at named interface points. They were previously called *widgets*.

== Introduction

An admin extension:

* Is contributed by an app and discovered by the host admin tool through a string identifier called an *interface*.
* Mounts only into tools that publish a matching interface name. The reserved name `"generic"` matches any tool.
* Is served by the admin engine through the `admin:extension` dispatcher API at `+/admin/<app>/<tool>/_/admin:extension/<app>:<extension-name>+`.
* Returns whatever HTTP payload the host tool expects — typically an HTML fragment or JSON.

Content Studio is the main consumer of admin extensions in practice, with interfaces such as `contentstudio.menuitem` and `contentstudio.contextpanel`.

== Usage

To declare an extension, place a descriptor and a controller under `+src/main/resources/admin/extensions/<extension-name>/+`:

[source,files]
----
src/main/resources/admin/extensions/
  my-extension/
    my-extension.yaml   # <1>
    my-extension.js     # <2>
    my-extension.svg    # <3>
----

<1> Descriptor — see <<#descriptor,below>>.
<2> HTTP controller for the extension URL.
<3> Optional SVG icon, served by the discovery endpoint.

The folder name, descriptor base name, and controller base name must all match `<extension-name>`.

[#descriptor]
== Descriptor

The descriptor is YAML only — `.yaml` and `.yml` are both accepted.

.src/main/resources/admin/extensions/<extension-name>/<extension-name>.yaml
[source,yaml]
----
kind: "AdminExtension"
title:                                # <1>
  text: "My extension"
  i18n: "i18n.my-extension.title"
description: "My extension"           # <2>
interfaces:                           # <3>
  - "contentstudio.contextpanel"
  - "com.example.myinterface"
allow:                                # <4>
  - "role:system.authenticated"
config:                               # <5>
  property_1: "value_1"
----

<1> *title* — string, or `{ text, i18n }` for localization. Used wherever the host tool renders an extension list.
<2> *description* — string, or `{ text, i18n }`.
<3> *interfaces* — names of interface points the extension plugs into. The host tool must publish a matching name in its own `interfaces:` list. Use the reserved value `"generic"` to make the extension mountable from any admin tool.
<4> *allow* — principal keys granted access. `role:system.admin` is always permitted in addition to anything listed here. Required to expose the extension to non-admins.
<5> *config* — free-form key/value map readable from the controller and surfaced in the discovery endpoint.

== Controller

The controller is a standard <<framework/http#,HTTP controller>> at `+src/main/resources/admin/extensions/<extension-name>/<extension-name>.js+`. The shape of the response is determined by the host tool — Content Studio context-panel extensions, for example, return an HTML fragment.

.Sample my-extension.js
[source,javascript]
----
exports.get = function (req) {
    const contentId = req.params.contentId;
    return {
        body: '<div>Inspecting content ' + contentId + '</div>',
        contentType: 'text/html'
    };
};
----

== URL

Extensions are served as service points under whichever admin tool hosts them, through the `admin:extension` dispatcher API:

* Invocation: `+/admin/<app>/<tool>/_/admin:extension/<app>:<extension-name>+`
* Discovery (list): `+/admin/<app>/<tool>/_/admin:extension?interface=<name>+`
* Discovery (icon): `+/admin/<app>/<tool>/_/admin:extension?icon&app=<app>&extension=<name>+`

For the dispatcher to be reachable, the host tool's descriptor must include `"admin:extension"` in its `apis:` list.

The interface match rule applies on every invocation: unless the extension's `interfaces:` list contains `"generic"`, at least one of its interfaces must also be present in the host tool's `interfaces:` list, otherwise the request returns *404 Not Found*.

== Access control

Two checks are enforced:

. The caller must hold `role:system.admin.login`.
. The extension's descriptor must permit the caller — `role:system.admin` is always permitted; otherwise the caller must match at least one principal in `allow:`.

== Discovery

The host tool typically queries `+?interface=<name>+` to find which extensions are available to the current user at a given interface point. The response is a JSON array, one entry per allowed extension:

[source,json]
----
[
  {
    "key": "com.example.app:my-extension",
    "title": "My extension",
    "description": "My extension",
    "iconUrl": "/admin/.../_/admin:extension?icon&app=com.example.app&extension=my-extension&v=...",
    "url": "/admin/.../_/admin:extension/com.example.app:my-extension",
    "interfaces": ["contentstudio.contextpanel"],
    "config": { "property_1": "value_1" }
  }
]
----

The listing is filtered by the caller's principals — extensions the caller cannot access are omitted.

== Interfaces

Interface names are free-form strings, not a closed enum. The contract is purely a string match between the extension's `interfaces:` and the host tool's `interfaces:`. To define a new interface point in your own admin tool, declare a stable name (e.g. `<app-key>.<purpose>`) in the tool's descriptor and document the request/response shape your tool expects from extensions.

Interfaces consumed in practice by Content Studio include `contentstudio.menuitem` and `contentstudio.contextpanel` — see Content Studio docs for the full list and the response shape each one expects.
