The Enonic Development Kit (EDK) is the server-side TypeScript/JavaScript framework Enonic XP exposes to app developers. 

Write your code in TypeScript, deploy it as an <<fundamentals/apps#, Enonic app>>, and get full access to XP's persistence, clustering, identity, scheduling, and HTTP machinery — powered by the <<fundamentals/runtime#, XP runtime>>, without Node.js, without a separate JS process, and with cluster-aware semantics out of the box.

This documentation covers the app-development surface only, other relevant documentation includes:

* https://developer.enonic.com/docs/cms[CMS docs] for content modelling and content APIs
* https://developer.enonic.com/docs/content-studio[Content Studio docs] for the content authoring interface
* https://developer.enonic.com/docs/xp[XP platform docs] for platform features and architecture

== Introduction

XP is highly modular: a single instance hosts many apps side by side. Each app is a self-contained bundle that can contribute <<examples/schemas#, content schemas>>, custom <<apis#, APIs>>, <<admin#, admin tools and extensions>>, <<http/id-providers#, ID providers>>, <<webapps#, webapps>>, or full <<sites#, websites>> — all powered by the same JS/TS runtime and the same standard <<libraries#, libraries>>.

== Where to start

<<tooling#, Tooling>>:: The CLI, sandboxes, starters, build system, and CI/CD setup that get you building quickly.

<<fundamentals#, Fundamentals>>:: The runtime, the app model, the module system, lifecycle hooks, and the cluster-wide event bus.

<<http#, HTTP>>:: The shared request pipeline, primitives (request/response, filters, error handlers, websockets, SSE), ID providers, and the four services XP dispatches to.

<<apis#, APIs>>, <<admin#, Admin>>, <<sites#, Sites>>, <<webapps#, Webapps>>:: The four services exposed by XP — custom APIs, admin UIs and extensions, CMS-driven sites, and standalone web applications.

<<background#, Background>>:: Tasks and scheduled jobs for asynchronous work.

<<libraries#, Libraries>>:: The platform's standard libraries — content, repos, identity, mail, and more.

<<examples#, Examples>>:: End-to-end examples for common scenarios.
